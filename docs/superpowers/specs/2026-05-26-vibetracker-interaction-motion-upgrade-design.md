# VibeTracker 交互动效升级设计

日期：2026-05-26

## 设计定位

本次升级的目标是把 VibeTracker 从“漂亮地展示项目”推进到“能感知项目正在前进的创作空间”。

动效关键词是：进展生命感、创作仪式感、Apple 式克制。

软件可以更动感，但动感必须来自用户动作和项目变化，而不是背景装饰或无意义循环。每一次 commit、封面更新、状态完成，都应该让用户感觉项目被认真记录了一次；普通浏览、筛选、编辑则保持轻巧、跟手、不打扰。

## 核心原则

1. **进展事件是主角**
   - 最高规格动效只给新增 commit、完成状态、设置封面、首次导入演示数据等高价值事件。
   - 普通 hover、筛选、输入、保存只使用短反馈。
   - 不做撒花、弹跳、旋转、大范围漂移。

2. **动效要有来源**
   - 新 commit 从输入区域附近进入时间线。
   - 编辑面板从对应 commit 附近展开。
   - 项目卡片进入详情时要保留空间来源感。
   - 任何弹层都不能像凭空出现。

3. **短、准、可解释**
   - 用户能理解动效发生的原因。
   - 完整 commit 仪式控制在约 1 秒内。
   - 微交互反馈控制在 80-180ms。
   - 页面与面板转场控制在 220-360ms。

4. **安静落位**
   - 动效结束后页面必须回到稳定、沉静、可长期使用的状态。
   - 不保留持续闪烁、强呼吸、大面积流光。
   - 背景环境动效只能非常低频、低透明度。

## 动效分层

### 1. 跟手反馈层

用于按钮、卡片、输入框、标签、图片缩略图。

推荐表现：

- 按钮按下：`scale(0.98)`，亮度轻微下降，80-120ms 回弹。
- 卡片 hover：`translateY(-3px)` 到 `translateY(-5px)`，边框变亮，阴影加深。
- 输入聚焦：边框与内阴影柔和点亮，避免强蓝光。
- 标签 hover：背景明度轻升，文字保持稳定。

这一层负责让软件“跟手”，不负责制造仪式感。

### 2. 空间连续层

用于页面切换、详情进入、侧边编辑面板、图片查看。

推荐表现：

- 画廊到详情：详情页顶部从卡片方向轻微放大进入，卡片列表淡退或错峰回位。
- 详情返回画廊：当前项目卡片优先清晰，其余卡片轻微错峰出现。
- 设置与标签页：像系统面板切换，使用短距离滑入和淡入。
- 编辑面板：从对应 commit 右侧或下方展开，背景轻微降层。

这一层负责让界面像一个连续空间，而不是多个硬切页面。

### 3. 进展仪式层

用于高价值动作。第一版最重要的是新增 commit。

完整 commit 仪式：

1. **输入确认，0-160ms**
   - 提交按钮按下。
   - 表单进入确认中状态。
   - 当前焦点光停住，避免表单突兀消失。

2. **时间线点亮，160-420ms**
   - 新 commit 从输入区域附近淡入并轻微上移到时间线。
   - 新时间线节点亮起一次。
   - 节点外圈出现极淡扩散环，只播放一次。
   - 下方旧节点轻微让位，不做大幅跳动。

3. **数据同步，420-720ms**
   - 当天热力图格子加深，并做一次很轻的脉冲。
   - 项目卡片最近 commit 平滑替换。
   - 总览页近期动态插入新记录。
   - 相关统计数字可以做短 count-up。

4. **沉静落位，720-980ms**
   - 如果 commit 图片成为封面候选，封面轻扫光一次。
   - 所有元素回到稳定状态。
   - 背景环境光恢复低频，不留下持续动效。

## 关键场景设计

### 新增 Commit

新增 commit 是 VibeTracker 的核心动作，也是本次升级的主舞台。

要求：

- 提交成功前不要播放成功仪式。
- 保存失败时使用错误状态，不播放点亮、热力图脉冲或封面扫光。
- 新节点应能被用户立刻定位到。
- 如果用户开启减少动态，只保留淡入和颜色变化。

### Commit 编辑

编辑 commit 不应等同于新增 commit。

要求：

- 打开编辑面板时，面板从该 commit 附近展开。
- 保存编辑后，该 commit 卡片轻微闪亮或边框点亮一次。
- 删除 commit 时，卡片收缩并淡出，时间线平滑补位。
- 删除后热力图、最近 commit、封面候选同步更新，但不播放完整新增仪式。

### 热力图

热力图是进展生命感的可视化核心。

要求：

- 当天新增 commit 后，对应格子颜色加深。
- 同一天多次提交，颜色阶梯变化要可见。
- 格子脉冲只在数据变化时触发一次。
- hover 仍保持安静，只显示日期与数量，不做强动画。

### 项目画廊

项目画廊需要体现最近活跃，而不是静态卡片墙。

要求：

- 最近更新的项目卡片可以有一次轻微强调。
- 卡片 hover 时封面做低幅缩放，最近 commit 区域更清晰。
- 图片封面加载时淡入，失败时平滑降级到文字封面。
- 没有图片的文字卡片不生成装饰封面。

### 总览页

总览页回答“最近什么东西在动”。

要求：

- 近期 commit 流新增记录时淡入置顶。
- 活跃项目列表可以按更新时间平滑重排。
- 统计数字变化时做短 count-up。
- 不把总览页做成强报表动画。

### 设置与状态管理

设置页保持系统感。

要求：

- 状态颜色修改时，色块、状态胶囊、使用该状态的预览同步过渡。
- 排序操作可以使用轻微位移反馈。
- 删除保护使用清晰的行内状态，不用夸张警告动画。

## Motion Token 建议

第一版建议先用 CSS 变量沉淀动效令牌。

```css
:root {
  --motion-fast: 120ms;
  --motion-base: 180ms;
  --motion-panel: 280ms;
  --motion-page: 340ms;
  --motion-ritual: 980ms;

  --ease-standard: cubic-bezier(0.2, 0, 0, 1);
  --ease-emphasized: cubic-bezier(0.16, 1, 0.3, 1);
  --ease-press: cubic-bezier(0.3, 0, 0.2, 1);

  --lift-card: -4px;
  --press-scale: 0.98;
}
```

使用规则：

- 所有页面与组件优先引用 token，不在页面里散写时长。
- 只在确实需要时增加新的 token。
- 第一阶段以 CSS 和 React 状态实现，不急于引入动画库。
- 只有跨页面共享元素转场需求明显时，再评估 `framer-motion`。

## 组件与工程边界

建议新增或整理这些小型基础能力：

- `AnimatedPage`：统一页面进入和离开基调。
- `MotionCard`：统一 hover、press、焦点、更新强调。
- `PresenceList`：用于 commit 流、项目卡片、最近动态的插入和移除动画。
- `RitualFeedback`：承载新增 commit 后的一次性点亮状态。
- `motion.ts`：集中定义动效阶段、延迟、CSS class 组合。

这些组件应保持轻量，不要建立过重的设计系统。当前阶段的目标是让现有页面动得一致、漂亮、可维护。

## 降级与可访问性

必须完整支持 `prefers-reduced-motion`。

减少动态时：

- 取消位移、缩放、扩散环、扫光。
- 保留透明度、颜色、边框变化。
- commit 新增仍应可识别，但不做空间移动。
- 背景环境动画停止。

性能要求：

- 不使用大面积 canvas 背景。
- 不做持续高频动画。
- 阴影、模糊、扫光只在少量元素上触发。
- 动画不能阻塞数据保存和页面交互。

## 实施建议

推荐分阶段实现：

1. **建立动效令牌与基础类**
   - 统一时长、缓动、press、hover、focus、reduced-motion。

2. **升级页面与导航转场**
   - 让画廊、详情、总览、设置拥有一致进出场。

3. **实现 commit 仪式**
   - 新增 commit 后联动时间线、热力图、项目卡片和总览流。

4. **打磨卡片与图片动效**
   - 项目卡片、封面图、截图缩略图、文字封面统一手感。

5. **完善编辑与状态管理微交互**
   - 编辑面板、删除、状态切换、颜色更新、排序反馈。

6. **验证与节制**
   - 检查是否过度动效。
   - 检查减少动态。
   - 检查 Electron 内真实交互手感。

## 验收标准

- 新增 commit 时，用户能明显感觉项目推进了一步。
- 动效主要围绕 commit、时间线、热力图和最近动态，不喧宾夺主。
- 页面切换有空间连续感，不再像硬切静态页面。
- 卡片、按钮、输入框手感统一且跟手。
- 背景有轻微空气感，但不会抢内容。
- `prefers-reduced-motion` 下仍可正常使用。
- 整体仍符合 Calm Apple Dark UI：冷静、高级、长期耐看。

## 不做的内容

第一版不做以下内容：

- 不做游戏化庆祝动画。
- 不做持续粒子、烟花、弹跳或旋转。
- 不引入重型动画库作为默认方案。
- 不重做信息架构。
- 不把 Prompt 模块重新做重。
- 不让动画改变数据含义或掩盖保存失败。

## 维护规则

后续新增交互前先检查：

- 这个动效是否来自用户动作或数据变化？
- 用户是否能理解它为什么发生？
- 它是否在 1 秒内安静落位？
- 减少动态时是否仍可使用？
- 它是否强化“项目进展跟进”主线？

如果答案是否定的，不应加入该动效。
