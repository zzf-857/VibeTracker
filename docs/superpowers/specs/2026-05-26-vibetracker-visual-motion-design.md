# DevTracker 视觉与动效设计基准

日期：2026-05-26

## 设计定位

DevTracker 的设计定位是：沉静的项目进展画廊。

它不是传统项目管理后台，也不是重型 Prompt 知识库。它应该像一个长期打开也舒服的本地创作空间：每个 vibecoding 项目像一张有状态的作品卡片，点进详情后能看到项目概览、最近提交、进展时间线、活跃热力图和成果截图。

后续所有页面和功能都必须服从这个气质：安静、漂亮、可持续跟进项目进展。

## 核心视觉原则

1. **像作品画廊，不像后台系统**
   - 外层项目页应优先呈现项目封面、项目名、状态和最近 commit。
   - 避免表格化、密集控件、硬边框和后台管理感。
   - 信息密度要适合日常扫视，但不能牺牲展示感。

2. **深色、克制、长期耐看**
   - 主背景使用接近黑色的深灰，不使用纯黑。
   - 面板使用柔和半透明灰，通过明度和边框区分层级。
   - 大面积避免高饱和颜色，颜色只用于状态、焦点和数据。

3. **Apple 式冷静高级感**
   - 页面要有充足留白和清晰层级。
   - 字体、圆角、阴影、面板都应克制。
   - 不做炫目的大渐变、装饰性光斑或过度科技风。

4. **进展优先**
   - 项目状态主要由最近 commit、时间线和热力图表达。
   - 不强调精确百分比。
   - 功能设计不能抢走“项目进展跟进”的主线。

## 色彩规范

基础色：

- 背景：`#080A0D`，深而不死黑。
- 主文字：接近白色，建议 `#F7F8FB`。
- 次级文字：冷灰，建议 `#A8B0BD`。
- 弱文字：建议 `#707A8A`。
- 分隔线：白色 10%-14% 透明度。
- 面板：白色 5%-11% 透明度。

功能色：

- 蓝色：主操作、焦点、当前选择。少量使用。
- 绿色：活跃、完成、热力图高活跃格。
- 琥珀色：暂停、提醒、待处理。
- 红色：仅用于危险操作，不用于普通强调。

禁止事项：

- 不使用大面积紫蓝渐变作为主视觉。
- 不使用装饰性光球、漂浮色块、花哨背景。
- 不用颜色替代清晰的信息层级。

## 字体规范

字体方向：Apple 式冷静高级感。

主字体：

```css
font-family:
  -apple-system,
  BlinkMacSystemFont,
  "Segoe UI",
  "Microsoft YaHei UI",
  "Microsoft YaHei",
  sans-serif;
```

等宽字体：

```css
font-family:
  "JetBrains Mono",
  "SF Mono",
  Consolas,
  monospace;
```

使用规则：

- 正文使用 400-500 字重。
- 标题使用 600-700 字重。
- 不使用负字距。
- 不用过粗字重制造“科技感”。
- commit 元信息、路径、时间、数字、代码片段使用 JetBrains Mono。

## 布局规范

### 外层项目页

外层项目页应像项目画廊。

项目卡片默认展示：

- 项目封面，如果项目已有截图。
- 项目名称。
- 自定义状态。
- 最近一条 commit。
- 少量标签或技术标识。

封面规则：

- 有截图时，默认自动选一张图作为封面。
- 后期允许手动设置封面图。
- 没有截图时，不自动生成装饰图，只显示文字型项目卡片。

### 项目详情页

详情页采用混合布局：

- 顶部是项目概览：项目名、简介、状态、最近提交、封面/截图、成果链接。
- 下方马上进入进展系统：commit 时间线、活跃热力图、提交编辑入口。
- 第一屏必须露出时间线开头，不能让顶部概览占满整个屏幕。

### 页面层级

- 主面板圆角建议 24-32px。
- 项目卡片圆角建议 22-28px。
- 按钮和标签使用胶囊形。
- 页面 section 不要伪装成浮动卡片。
- 避免卡片套卡片，除非是明确的重复项或编辑面板。

## 动效规范

动效原则：轻、短、可解释。

时长建议：

- hover / press：120-180ms。
- 小组件展开：180-240ms。
- 页面进入 / 详情展开：220-320ms。
- 数据替换：160-240ms。

缓动建议：

- 默认使用自然的 ease-out。
- 不使用夸张弹跳。
- 不使用旋转、大幅漂移或花哨动效。

关键动效：

1. **项目卡片进入详情页**
   - 应有从卡片自然展开到详情页的来源感。
   - 不要硬切页面。

2. **新增 commit**
   - 新时间线节点淡入。
   - 后续节点轻微让位。
   - 热力图当天格子轻微点亮。
   - 外层项目卡片的最近 commit 平滑替换。

3. **提交编辑面板**
   - 从对应 commit 附近展开。
   - 第一版可用侧边面板或内联展开。
   - 避免无来源感的突兀弹窗。

4. **状态变化**
   - 状态标签颜色和文字平滑过渡。
   - 不做闪烁和大面积动效。

5. **减少动态**
   - 尊重系统 `prefers-reduced-motion`。
   - 减少动态时保留透明度变化，取消位移和缩放。

## 组件方向

### 按钮

- 主按钮使用浅色填充或柔和蓝色，数量要少。
- 常规操作使用低对比按钮。
- 危险操作使用红色，但只在确认区域出现。

### 标签

- 标签使用胶囊形。
- 颜色要柔和，不要像彩色贴纸一样抢视觉。
- 状态标签和分类标签要有不同视觉层级。

### 热力图

- 热力图按 commit 数量计算活跃度。
- 颜色从暗灰到绿色渐进。
- 不把进度变化量混入热力图计算。

### Commit 时间线

- 时间线是项目详情页核心组件。
- 每条 commit 默认包含标题、描述、时间。
- 支持后续进入独立编辑面板补截图、链接和更多信息。
- 时间线视觉应安静清晰，不做过度 Git 拟物化。

## 功能边界提醒

当前阶段先维护视觉与动效基准，功能规划后续再细化。

已确认的功能方向只作为设计背景：

- 项目主线是可视化介绍和跟进 vibecoding 项目进度。
- Commit 式进展记录是核心交互。
- Prompt 速记是轻量辅助功能，不做重型 Prompt 库。
- 项目状态支持自定义。
- 不以精确百分比作为项目进度主表达。

## 后续维护规则

每次新增页面或组件前，需要先检查是否符合本设计基准：

- 是否仍像项目进展画廊？
- 是否避免后台管理感？
- 是否保持 Apple 式冷静高级感？
- 动效是否有来源、有意义、不过度？
- 是否把项目进展放在了主线位置？

如果某个功能需要破坏这些规则，必须先更新本设计文档，再进入实现。
