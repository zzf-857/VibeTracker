# Calm Apple Dark UI 设计范式

日期：2026-05-26

## 设计定位

这套设计范式适合长期打开、需要承载创作过程、项目进度、知识沉淀或本地工具流的软件。

它的关键词是：沉静、清晰、高级、可持续使用。

它不是营销型页面，不靠巨大标题、强烈渐变或装饰图形制造第一眼冲击；也不是传统后台，不用表格、硬边框和密集控件堆出管理感。它更像一个 Apple 式的本地工作空间：内容是主角，界面负责把信息托起来。

## 核心原则

1. **像作品空间，不像后台系统**
   - 第一屏应该让用户看到正在处理的对象，而不是功能说明。
   - 卡片、封面、状态、最近动态比统计表更重要。
   - 避免大面积表格、纯列表、密集按钮和强管理台气质。

2. **深色、冷静、长期耐看**
   - 背景接近黑色但不纯黑。
   - 面板用半透明灰和细边框区分层级。
   - 色彩只用于状态、焦点、数据和少量引导。

3. **Apple 式高级感来自克制**
   - 留白、对齐、层级、字体权重比装饰更重要。
   - 动效短、轻、有来源感。
   - 组件要像系统控件，而不是网页装饰物。

4. **信息以“状态变化”组织**
   - 页面不只是展示数据，还要表达对象最近发生了什么。
   - 最近记录、时间线、热力图、状态标签是核心表达方式。
   - 精确百分比、复杂指标、硬核图表只在确实必要时出现。

## 色彩系统

### 基础色

```css
:root {
  --bg-primary: #080A0D;
  --bg-secondary: rgba(255, 255, 255, 0.07);
  --bg-tertiary: rgba(255, 255, 255, 0.105);
  --bg-sidebar: rgba(255, 255, 255, 0.045);

  --text-primary: #F7F8FB;
  --text-secondary: #A8B0BD;
  --text-tertiary: #707A8A;

  --border-primary: rgba(255, 255, 255, 0.12);
  --border-subtle: rgba(255, 255, 255, 0.08);
}
```

### 功能色

```css
:root {
  --accent-blue: #74A9FF;
  --accent-purple: #BC8CFF;
  --accent-orange: #F3BB6C;
  --accent-red: #FF6B6B;
  --status-completed: #63D693;
}
```

### 使用规则

- 蓝色用于当前选择、焦点、主路径。
- 绿色用于完成、可演示、活跃。
- 紫色用于设计打磨、创意、动效。
- 琥珀色用于暂停、待处理、提醒。
- 红色只用于危险操作。
- 不使用大面积紫蓝渐变作为主视觉。
- 不使用高饱和背景、霓虹光效、装饰性光球。

## 字体系统

### 主字体

```css
font-family:
  -apple-system,
  BlinkMacSystemFont,
  "Segoe UI",
  "Microsoft YaHei UI",
  "Microsoft YaHei",
  sans-serif;
```

### 等宽字体

```css
font-family:
  "JetBrains Mono",
  "SF Mono",
  Consolas,
  monospace;
```

### 使用规则

- 中文正文用 400-500 字重。
- 页面标题用 600-700 字重，不要过粗。
- 不使用负字距。
- 时间、路径、commit、数字、代码片段使用等宽字体。
- 按钮和标签文字要短，避免把说明文字塞进控件。

## 背景与空间

### 背景

背景应是深色空间，不是装饰画布。

推荐结构：

```css
#root {
  background:
    radial-gradient(circle at 16% 0%, rgba(116, 169, 255, 0.16), transparent 30%),
    radial-gradient(circle at 82% 8%, rgba(99, 214, 147, 0.10), transparent 25%),
    linear-gradient(180deg, #101318, #080A0D 58%);
}
```

环境光可以非常慢地漂移，但必须克制：

- 透明度低。
- 模糊柔和。
- 速度慢。
- 不形成可见装饰物。
- 不抢内容层级。

### 布局

- 主内容最大宽度建议 `1280px-1440px`。
- 页面外边距建议 `32px-40px`。
- 面板间距建议 `20px-28px`。
- 首屏必须让核心内容露出来，不要让顶部介绍占满整屏。
- 固定导航区域要窄、稳定、少文字，以图标为主。

## 面板与卡片

### Glass Panel

```css
.glass-panel {
  background: rgba(255, 255, 255, 0.07);
  border: 1px solid rgba(255, 255, 255, 0.12);
  box-shadow: 0 28px 80px rgba(0, 0, 0, 0.28);
  backdrop-filter: blur(18px);
  position: relative;
  overflow: hidden;
}
```

### 圆角

- 主面板：`28px-32px`
- 内容卡片：`22px-30px`
- 输入框、按钮、标签：胶囊形或 `16px-24px`
- 小图、截图缩略图：`14px-18px`

### 层级

- 页面 section 不要伪装成一堆浮动卡片。
- 只有独立对象、重复项目、编辑面板适合用卡片。
- 避免卡片套卡片；如果必须嵌套，内层要弱化阴影和边框。

## 组件范式

### 按钮

按钮分三类：

- 主按钮：浅色填充，数量极少。
- 次级按钮：半透明灰底，细边框。
- 危险按钮：红色，只在确认或删除区域出现。

按钮应该：

- 使用图标加短文字。
- 高度稳定，通常 `36px-48px`。
- hover 只改变明度、边框或轻微位移。
- 不写长句说明。

### 输入与编辑入口

推荐使用 **Inline Composer**：

- 默认只显示一个轻量入口按钮。
- 点击后在当前区域展开编辑面板。
- 避免右上角堆一张厚重表单卡片。
- 输入框使用深色半透明背景，不使用浏览器默认白底。
- 复杂编辑可以放到侧边面板，但要有来源方向。

### 标签与状态

- 标签使用胶囊形。
- 状态标签可以带柔和背景色，例如 `${color}18`。
- 状态点可以轻微呼吸，但不能闪烁。
- 分类标签和状态标签要区分层级：状态更强调，分类更低调。

### 图片与封面

- 图片是内容，不是装饰。
- 有真实截图时优先使用真实截图。
- 没有图片时显示文字型卡片，不自动生成花哨占位图。
- 本地图片要通过安全读取通道转成可显示资源，不直接依赖 `file://`。
- 图片失效时显示安静占位，不阻塞页面。

### 时间线

时间线适合表达持续推进的产品：

- 每条记录有标题、描述、时间。
- 记录可附带截图。
- 时间线节点使用轻微状态点。
- 新记录淡入，不做夸张跳动。
- 不要过度拟物化 Git；借用 commit 心智即可。

### 热力图

热力图适合表达活跃节奏：

- 按事件数量计算，不混入进度百分比。
- 颜色从暗灰到绿色渐进。
- 方格小、安静、可扫视。
- hover title 可以显示日期和数量。

## 动效系统

动效基调：轻、短、有来源。

### 时间建议

- hover / press：`120ms-180ms`
- 页面进入：`220ms-320ms`
- 小面板展开：`180ms-240ms`
- 侧边编辑面板：`220ms-280ms`
- 数据替换：`160ms-240ms`

### 推荐动效类

```css
.page-enter {
  animation: pageEnter 260ms ease-out both;
}

.stagger-item {
  animation: itemRise 320ms ease-out both;
  animation-delay: calc(var(--stagger, 0) * 55ms);
}

.motion-card {
  transition:
    transform 180ms ease-out,
    border-color 180ms ease-out,
    background-color 180ms ease-out,
    box-shadow 180ms ease-out,
    opacity 180ms ease-out;
}
```

### 动效规则

- 页面进入只做淡入和 8-10px 位移。
- 卡片 hover 最多 `translateY(-4px)`。
- 面板不要弹跳、旋转、大幅缩放。
- 列表可以错峰，但延迟总时长不要拖沓。
- 背景呼吸要慢，像环境光，不像动画特效。
- 必须支持 `prefers-reduced-motion`。

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 1ms !important;
    animation-iteration-count: 1 !important;
    transition-property: opacity, color, background-color, border-color !important;
    transition-duration: 1ms !important;
    transform: none !important;
  }
}
```

## 页面模式

### 总览页

适合结构：

- 顶部：页面标题 + 一句说明 + 主入口按钮。
- 中部：4 个以内关键统计。
- 主区域：最近活跃对象。
- 侧栏：最近动态、状态分布、轻量热力图。

总览页不应变成报表页。它应该回答：最近什么东西在动？

### 画廊页

适合结构：

- 顶部：标题 + 新建入口。
- 过滤条：标签、搜索。
- 主体：对象卡片网格。
- 卡片：封面、名称、状态、简介、最近记录、标签或路径。

画廊页不应使用表格作为第一视图。

### 详情页

适合结构：

- 顶部：对象档案 + 封面。
- 中部：核心推进系统，如时间线。
- 侧栏：热力图、截图、相关链接。
- 编辑入口：内联或侧边面板。

详情页第一屏必须露出“下一层内容”，不能被顶部简介完全占满。

### 设置页

设置页也要保持产品气质：

- 分组清楚。
- 每个设置项像一个可编辑对象。
- 避免原生下拉和粗糙表格。
- 颜色选择用色块。
- 删除保护用行内确认，不只依赖 alert。

## 真实数据与演示数据

为了展示完整形态，可以提供演示数据，但要遵守边界：

- 演示数据可以真实落库，但必须使用固定 id，方便重复导入。
- 不覆盖用户自建项目。
- 演示截图应是项目资产或用户可访问的本地路径。
- 空状态可以用 mock 展示，但必须明确标注。
- 一旦用户有真实数据，界面应优先呈现真实数据。

## 不适合这套范式的场景

这套设计不适合：

- 强营销落地页。
- 高密度财务报表。
- 需要大量表格批量编辑的后台。
- 游戏化、强视觉刺激产品。
- 面向极低性能设备且无法承担 blur/阴影成本的场景。

如果要用于这些场景，需要先降低玻璃效果、提高密度或重做信息架构。

## 落地检查清单

开始设计新软件前检查：

- 这个软件的“核心对象”是什么？
- 用户打开后最想扫一眼什么状态？
- 第一屏是否直接展示核心对象，而不是说明文案？
- 是否能用卡片、时间线、状态、热力图表达变化？
- 是否避免了后台管理感？
- 是否有真实内容或可信演示数据支撑视觉？

实现前检查：

- 背景是否深而不死黑？
- 面板是否用半透明和细边框建立层级？
- 字体是否使用系统字体栈？
- 路径、时间、数字是否使用等宽字体？
- 主按钮是否足够少？
- 空状态是否干净，不乱生成装饰？
- 图片失效是否有安静降级？

验收前检查：

- 页面进入是否柔和？
- 卡片 hover 是否克制？
- 动效是否支持减少动态？
- 文本是否没有溢出？
- 是否没有卡片套卡片造成噪音？
- 页面是否能长期打开而不累？

## 代码落地建议

在新项目中优先建立这些基础：

- `:root` 颜色 token。
- `glass-panel`。
- `page-enter`。
- `motion-card`。
- `stagger-item`。
- `breathing-dot`。
- `prefers-reduced-motion` 降级。
- `SafeImage` 或等价的图片降级组件。

不要一开始就写完整设计系统。先把这些低层 token 和常用动效类建立好，再随着真实页面沉淀组件。
